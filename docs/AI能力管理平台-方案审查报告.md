# AI能力管理平台 — 技术方案审查报告

## 审查结论

方案整体架构清晰，四层设计（LLM→MCP→Skill→Agent）逻辑自洽，但存在以下**8个待补充点**和**5个需优化点**。

---

## 一、待补充内容

### 1.1 缺少：SDK详细设计与API接口规范

**问题：** 方案提到了`@your-org/ai-platform-sdk`，但缺少完整的API接口规范文档。业务系统如何接入、SDK暴露哪些方法、错误码定义、限流策略等均未明确。

**建议补充：**
- REST API完整路由表（如 `POST /api/v1/skills/:slug/run`）
- SDK方法签名与TypeScript类型定义
- 错误码体系（如 `SKILL_NOT_FOUND`、`MODEL_QUOTA_EXCEEDED`、`AGENT_TIMEOUT`）
- 限流策略（按项目/按Skill/按模型的QPS限制）
- WebSocket接口规范（用于Agent实时进度推送和人工审核节点）

---

### 1.2 缺少：数据迁移与初始化方案

**问题：** 方案提到"批量导入87个Prompt"只用1天，但缺少具体的迁移脚本设计。

**建议补充：**
- 迁移脚本如何从代码中自动提取Prompt常量
- 如何映射现有的`invokeLLM`调用参数到Skill的inputSchema
- 如何处理嵌套在业务逻辑中的动态Prompt（如模板拼接）
- 迁移后的验证策略（对比新旧输出一致性）
- 回滚方案（迁移失败时如何恢复）

---

### 1.3 缺少：Agent暂停/恢复的状态持久化细节

**问题：** Agent执行到"人工审核"节点时暂停，但缺少以下细节：
- 暂停状态如何持久化（用户可能数小时后才回来确认）
- 多个用户同时运行同一Agent时的隔离
- 暂停超时处理（如24小时未确认自动取消）
- 恢复时如何重建执行上下文

**建议补充：**
- 暂停状态存储方案（数据库 + Redis缓存）
- 超时策略配置（每个Agent可设置不同超时）
- 恢复时的上下文重建逻辑
- 通知机制（暂停时通知用户，超时前提醒）

---

### 1.4 缺少：并发控制与队列设计

**问题：** 当多个项目同时调用大量Skill/Agent时，如何控制并发？方案中未涉及任务队列。

**建议补充：**
- 是否需要任务队列（如Bull/BullMQ）
- 按项目的并发限制（防止单项目占满资源）
- 按模型的并发限制（LLM API通常有rate limit）
- Agent执行的优先级队列
- 长时间运行Agent的超时与取消机制

---

### 1.5 缺少：版本管理的回滚粒度

**问题：** Skill版本管理提到了"一键回滚"，但缺少：
- 回滚是否影响正在运行的Agent（使用旧版Skill的Agent怎么办）
- 是否支持灰度发布（新版本只对部分流量生效）
- Agent引用Skill时是否锁定版本（pin version vs always-latest）

**建议补充：**
- 版本引用策略：`latest`（始终最新）vs `pinned`（锁定版本号）
- 灰度发布：按百分比或按项目灰度
- 回滚影响范围分析

---

### 1.6 缺少：安全性设计

**问题：** 方案中"自定义脚本"MCP类型存在安全风险，且跨项目API调用的安全模型不够详细。

**建议补充：**
- 自定义脚本的沙盒执行环境（vm2/isolated-vm）
- 脚本可访问的API白名单
- API Key的权限粒度（只读/执行/管理）
- 敏感数据（如API密钥）的加密存储
- 审计日志（谁在什么时候改了什么）

---

### 1.7 缺少：前端技术选型与组件库

**问题：** 方案提到React Flow用于Agent编排，Monaco Editor用于Prompt编辑，但缺少整体前端架构。

**建议补充：**
- 前端框架选型（是否复用现有React+Vite+Tailwind）
- 状态管理方案（Agent编排器的复杂状态）
- 组件库选择（shadcn/ui继续使用 or 其他）
- 代码编辑器配置（Monaco的语法高亮、自动补全）
- 可视化图表库（监控面板用Recharts/Echarts）

---

### 1.8 缺少：部署架构图

**问题：** 方案说"独立部署"，但缺少具体的部署拓扑。

**建议补充：**
- 部署在哪里（Manus WebDev / 独立服务器 / 云服务）
- 数据库是否与业务系统共享还是独立
- 是否需要Redis（缓存+队列）
- 域名规划（如 ai-platform.your-domain.com）
- 与业务系统的网络拓扑（同VPC / 公网API）

---

## 二、需优化内容

### 2.1 知识库检索方式过于简单

**当前方案：** 知识库检索支持 keyword / semantic / hybrid 三种，但未说明semantic搜索如何实现。

**优化建议：**
- 明确向量数据库选型（pgvector / Pinecone / 自建）
- Embedding模型选择（text-embedding-3-small等）
- 是否在Phase 1就需要向量搜索，还是先用keyword后续升级
- 建议Phase 1先做keyword搜索，Phase 5再升级为hybrid

---

### 2.2 Agent编排器复杂度可能超出预期

**当前方案：** 4天完成React Flow可视化编排器。

**优化建议：**
- React Flow的自定义节点开发（9种节点类型）+ 连线验证 + 表单配置面板，4天可能偏紧
- 建议拆为：基础画布+3种核心节点（2天）→ 剩余节点类型（2天）→ 高级功能如循环/并行（额外2天）
- 或者Phase 4.2调整为5-6天

---

### 2.3 自动学习机制缺少人工审核

**当前方案：** 高采纳率自动进入案例库，低采纳率自动进入错误模式库。

**优化建议：**
- 自动收集的案例应该有"待审核"状态，管理员确认后才正式生效
- 避免噪声数据污染知识库（用户可能因为其他原因不采纳，不代表AI输出有问题）
- 建议增加"候选池 → 人工审核 → 正式入库"的流程

---

### 2.4 MCP健康检查缺少告警机制

**当前方案：** 有健康监控状态显示，但缺少主动告警。

**优化建议：**
- MCP连续失败N次后自动告警（邮件/站内通知）
- 自动降级策略（MCP不可用时，Agent跳过该节点或使用缓存数据）
- 定时健康检查频率配置

---

### 2.5 工期估算偏乐观

**当前方案：** 31个工作日完成全部5个Phase。

**优化建议：**
- Agent可视化编排器（Phase 4.2）建议从4天调整为6天
- 知识库联动（Phase 5.1）如果包含向量搜索，建议从2天调整为4天
- 测试和联调时间未单独列出，建议每个Phase末尾预留1天集成测试
- **调整后总计：约35-38个工作日（7-8周）**

---

## 三、优先级建议

如果资源有限，建议按以下顺序补充：

| 优先级 | 补充项 | 理由 |
|--------|--------|------|
| P0 | SDK/API接口规范 | 业务系统接入的基础 |
| P0 | 部署架构图 | 影响技术选型和成本 |
| P1 | 并发控制设计 | 多项目场景必须 |
| P1 | Agent暂停/恢复细节 | 核心功能的可靠性 |
| P1 | 安全性设计 | 生产环境必须 |
| P2 | 数据迁移方案 | 可以在开发过程中细化 |
| P2 | 版本管理粒度 | 可以先做简单版 |
| P2 | 前端技术选型 | 可以沿用现有技术栈 |

## 四、总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | ★★★★☆ | 四层架构清晰，但缺少部署拓扑 |
| 功能完整性 | ★★★★☆ | 核心功能覆盖全面，细节待补充 |
| 可行性 | ★★★☆☆ | 工期偏乐观，Agent编排器复杂度高 |
| 扩展性 | ★★★★★ | 多项目+插件化设计，扩展性优秀 |
| 安全性 | ★★☆☆☆ | 安全设计不足，需要补充 |
| 实用性 | ★★★★★ | 解决了核心痛点（配置驱动代替开发驱动） |

**总结：** 方案方向正确，架构设计优秀，主要需要补充**接口规范、安全设计、并发控制**三个方面的细节，并适当调整工期估算。
