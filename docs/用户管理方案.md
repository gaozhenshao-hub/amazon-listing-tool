# 亚马逊运营AI提效工具 — 用户管理与独立部署方案

> **文档版本**：v3.0 | **日期**：2026-03-18
> 本方案基于两家公司独立部署、知识库双向同步的需求，设计完整的用户管理、角色权限、部署配置化和系统互联体系。

---

## 1. 需求概要

| 维度 | 需求描述 |
|------|---------|
| **部署模式** | 同一套代码，两次独立部署（各自独立URL、独立数据库） |
| **您的公司** | 约20人，使用领星ERP，作为主系统 |
| **朋友公司** | 约30人，使用赛狐ERP，独立部署后可自行二次开发 |
| **知识库** | 双向同步，两套系统都可上传，互相同步 |
| **使用量统计** | 各自统计，您可通过后台远程查看朋友系统的使用数据 |
| **角色体系** | 管理员 / 运营主管 / 运营专员 / 产品开发 / 财务 / 采购 / 美工（7种角色 + 超级管理员） |
| **登录方式** | 邮箱和手机号都支持，密码最少8位含大小写和数字 |
| **代码管理** | 通过GitHub共享代码库，朋友可Fork后自行修改优化 |

## 2. 整体架构

### 2.1 独立部署架构

```
┌─────────────────────────────┐     ┌─────────────────────────────┐
│     您的系统 (Instance A)     │     │   朋友系统 (Instance B)      │
│     xxx-a.manus.space        │     │   xxx-b.manus.space          │
│     独立数据库                │     │   独立数据库                  │
│     领星ERP                  │     │   赛狐ERP                    │
│     20人团队                 │     │   30人团队                    │
│                              │     │                              │
│  ┌─────────────────────┐    │     │    ┌─────────────────────┐   │
│  │     知识库数据        │◄───┼─────┼───►│     知识库数据        │   │
│  └─────────────────────┘    │     │    └─────────────────────┘   │
│                              │     │                              │
│  ┌─────────────────────┐    │     │    ┌─────────────────────┐   │
│  │   使用量统计面板      │◄───┼─────┼────│   使用量上报          │   │
│  │   (可查看B的数据)     │    │     │    │   (定期上报给A)       │   │
│  └─────────────────────┘    │     │    └─────────────────────┘   │
└─────────────────────────────┘     └─────────────────────────────┘
              │                                    │
              └──────────── GitHub ─────────────────┘
                        (共享代码库)
                   朋友可Fork后自行修改
```

### 2.2 系统不做多租户

由于两家公司独立部署，**不需要tenantId隔离**。每套系统内部是单公司使用，通过用户角色控制权限即可。

### 2.3 部署配置化

通过环境变量控制每套系统的差异化配置：

| 环境变量 | 说明 | 系统A示例 | 系统B示例 |
|---------|------|----------|----------|
| `COMPANY_NAME` | 公司名称 | 您的公司名 | 朋友公司名 |
| `COMPANY_LOGO` | 公司Logo URL | logo-a.png | logo-b.png |
| `ERP_TYPE` | ERP类型 | `lingxing` | `saihu` |
| `INSTANCE_ID` | 系统实例唯一标识 | `instance-a` | `instance-b` |
| `PEER_SYNC_ENABLED` | 是否启用知识库同步 | `true` | `true` |
| `PEER_API_URL` | 对方系统的API地址 | `https://xxx-b.manus.space` | `https://xxx-a.manus.space` |
| `PEER_API_KEY` | 系统间通信密钥 | 共享同一个密钥 | 共享同一个密钥 |
| `USAGE_REPORT_ENABLED` | 是否上报使用量 | `false`（主系统不上报） | `true`（向A上报） |
| `USAGE_REPORT_URL` | 使用量上报地址 | — | `https://xxx-a.manus.space` |

## 3. 角色权限体系

### 3.1 八种角色定义

## 5. 账号管理系统

### 5.1 登录体系

实现**独立的用户名/密码登录系统**，与现有Manus OAuth并行：

```
                  ┌──────────────────────────────────────────┐
                  │              统一登录页                    │
                  │  ┌────────────────┐  ┌────────────────┐  │
                  │  │ 邮箱/手机号登录  │  │  Manus OAuth   │  │
                  │  │ (团队成员)      │  │  (系统管理员)    │  │
                  │  └───────┬────────┘  └───────┬────────┘  │
                  └──────────┼───────────────────┼───────────┘
                             │                   │
                             └─────────┬─────────┘
                                       │
                               ┌───────▼───────┐
                               │  统一Session   │
                               │  (JWT Cookie)  │
                               └───────────────┘
```

| 项目 | 方案 |
|------|------|
| 密码存储 | bcrypt哈希，saltRounds=12 |
| 密码策略 | 最少8位，必须含大小写字母和数字 |
| 登录接口 | `POST /api/auth/login`（邮箱或手机号 + 密码） |
| Session | 复用现有JWT Cookie机制，payload增加`userId`和`role`字段 |
| 首次登录 | 管理员创建账号时设置初始密码，首次登录强制修改 |
| 密码重置 | 管理员可重置用户密码 |
| 登录保护 | 连续5次失败锁定15分钟 |
| 登录标识 | 支持邮箱或手机号登录（自动识别格式） |

## 6. 知识库双向同步（P2P）

### 6.1 同步架构

两套系统地位对等，都可以创建/编辑知识库内容，通过API双向同步：

```
┌──────────────┐                          ┌──────────────┐
│   系统A       │    知识库同步API           │   系统B       │
│              │                          │              │
│  创建内容 ────┼──── POST /api/sync/push ──┼──► 接收内容   │
│              │                          │              │
│  接收内容 ◄───┼──── POST /api/sync/push ──┼──── 创建内容  │
│              │                          │              │
│  查询变更 ────┼── GET /api/sync/changes ──┼──► 返回增量   │
│              │                          │              │
│  返回增量 ◄───┼── GET /api/sync/changes ──┼──── 查询变更  │
└──────────────┘                          └──────────────┘
```

### 6.2 同步策略

| 策略项 | 方案 |
|-------|------|
| **同步触发** | 定时同步（每30分钟）+ 手动触发 |
| **同步范围** | 仅同步`visibility = 'published'`的已发布内容 |
| **数据标识** | 每条数据有`originInstanceId`标记来源系统 |
| **冲突处理** | 以最后修改时间（`updatedAt`）为准，保留最新版本 |
| **删除同步** | 软删除标记同步，不物理删除远程数据 |
| **认证方式** | API Key认证（`PEER_API_KEY`环境变量） |
| **传输安全** | HTTPS + API Key Header |

### 6.3 同步API接口

```typescript
// 知识库同步API（Express路由，不走tRPC，因为是系统间通信）
// POST /api/sync/push — 推送本地变更到远程
{
  apiKey: string,
  instanceId: string,
  changes: [{
    resourceType: 'kb_product' | 'kb_listing' | 'kb_image_set' | 'kb_video' | 'kb_skill',
    action: 'create' | 'update' | 'delete',
    data: { ... },  // 完整数据
    originId: number,  // 来源系统中的ID
    updatedAt: number  // 时间戳
  }]
}

// GET /api/sync/changes?since=<timestamp>&type=<resourceType>
// 查询指定时间之后的变更列表

// POST /api/sync/usage-report — 使用量上报
{
  apiKey: string,
  instanceId: string,
  instanceName: string,
  date: string,  // YYYY-MM-DD
  stats: { totalUsers, activeUsers, aiCallCount, ... }
}
```

管理员可在"知识库同步"页面：

## 7. 使用量统计

### 7.1 本地统计

每套系统自动统计本地使用量：

| 统计项 | 说明 |
|-------|------|
| AI调用次数 | 每次LLM调用自动计数 |
| AI Token消耗 | 输入+输出Token总量 |
| 爬虫调用次数 | 亚马逊页面爬取次数 |
| 存储使用量 | S3文件存储总量 |
| API调用次数 | 所有API调用总量 |
| 活跃用户数 | 每日登录用户数 |

### 7.2 远程查看

您的系统（系统A）可以查看朋友系统（系统B）的使用量：

- 系统B通过`POST /api/sync/usage-report`每日自动上报使用数据
- 系统A将数据存入`remote_usage_snapshots`表
- 您在"使用量统计"页面可以切换查看本地/远程数据
- 支持Recharts图表展示趋势

## 8. 朋友公司独立部署指南

### 8.1 部署步骤

```
1. 朋友用自己的Manus账号登录
2. 从GitHub Fork代码仓库（可自行修改优化）
3. 在Manus上创建新的WebDev项目，导入Fork后的代码
4. 配置环境变量：
   - COMPANY_NAME = "朋友公司名"
   - ERP_TYPE = "saihu"
   - INSTANCE_ID = "instance-b"
   - PEER_SYNC_ENABLED = "true"
   - PEER_API_URL = "https://xxx-a.manus.space"
   - PEER_API_KEY = "共享密钥"
   - USAGE_REPORT_ENABLED = "true"
   - USAGE_REPORT_URL = "https://xxx-a.manus.space"
5. 运行数据库迁移
6. 创建管理员账号
7. 开始使用
```

### 8.2 朋友可自行修改的部分

| 可修改项 | 说明 |
|---------|------|
| UI主题/配色 | 修改index.css中的CSS变量 |
| 公司Logo/名称 | 通过环境变量配置 |
| ERP对接 | 实现赛狐ERP Adapter（代码中已预留接口） |
| 新增功能 | 在Fork的代码上自由开发 |
| 角色权限 | 可根据需要调整角色定义 |

### 8.3 代码同步策略

```
GitHub主仓库（您维护）
    │
    ├── 您的部署：直接使用主仓库代码
    │
```

## 10. ERP适配层设计

### 10.1 抽象接口

```typescript
// ERP适配层统一接口（领星和赛狐各自实现）
interface ERPAdapter {
  // 鉴权
  authenticate(): Promise<void>;
  refreshToken(): Promise<void>;
  
  // 统一数据输出格式
  getProducts(): Promise<StandardProduct[]>;
  getProfitReport(dateRange: DateRange): Promise<StandardProfitReport>;
  getAdCampaigns(dateRange: DateRange): Promise<StandardAdCampaign[]>;
  getInventory(): Promise<StandardInventoryItem[]>;
  getOrders(dateRange: DateRange): Promise<StandardOrder[]>;
  getSearchTerms(dateRange: DateRange): Promise<StandardSearchTerm[]>;
}

// 工厂模式：根据环境变量选择适配器
function createERPAdapter(): ERPAdapter {
  const erpType = process.env.ERP_TYPE || 'none';
  switch (erpType) {
    case 'lingxing': return new LingXingAdapter();
    case 'saihu': return new SaiHuAdapter();
    default: return new NoopAdapter();
  }
}
```

### 10.2 当前实现优先级

| 优先级 | 适配器 | 说明 |
|-------|--------|------|
| P0 | LingXingAdapter | 您的系统使用，优先实现 |
| P1 | SaiHuAdapter | 朋友系统使用，预留接口，后续实现 |
| P2 | NoopAdapter | 无ERP时的空实现，返回空数据 |
