# 属性标注与标签管理 — 现有逻辑全链路梳理

> 本文档完整梳理当前系统中"属性标注"和"标签管理"两大功能模块的数据结构、业务流程、前后端交互逻辑及其与下游分析模块的关联关系。

---

## 一、核心概念与数据模型

系统中存在**两套并行的标签体系**，分别服务于不同的业务场景：

| 体系 | 数据表 | 粒度 | 用途 | 触发方式 |
|------|--------|------|------|----------|
| **产品级标签（属性标注）** | `dev_product_tags` | 每个ASIN × 每个维度 = 一条记录 | 为每个竞品ASIN打上具体的属性值，用于全景分析表展示和筛选 | AI批量打标 / 全景表手动编辑 |
| **项目级标签（标签管理）** | `dev_project_tag_categories` + `dev_project_tag_items` | 项目级别的标签分类和标签值定义 | 定义本项目的属性维度框架和可选值，作为AI打标和交叉分析的基础 | AI生成 / 手动添加 / CSV导入 |

### 1.1 产品级标签表 `dev_product_tags`

该表存储每个ASIN在每个属性维度上的具体标签值，是"属性标注"功能的核心数据。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT (PK) | 自增主键 |
| `projectId` | INT | 所属项目ID |
| `asin` | VARCHAR(20) | 产品ASIN |
| `dimensionName` | VARCHAR(100) | 属性维度名称（如"材质"、"功能"） |
| `dimensionValue` | VARCHAR(255) | 该ASIN在此维度的具体值（如"不锈钢"、"防水"） |
| `source` | ENUM('ai','manual','specification') | 标签来源 |
| `confirmed` | INT | 是否已确认（0=未确认，1=已确认） |
| `createdAt` | TIMESTAMP | 创建时间 |

**数据示例：**

| projectId | asin | dimensionName | dimensionValue | source |
|-----------|------|---------------|----------------|--------|
| 60002 | B0ABC123 | 材质 | 不锈钢 | ai |
| 60002 | B0ABC123 | 功能 | 防水 | manual |
| 60002 | B0DEF456 | 材质 | 铝合金 | ai |

### 1.2 项目级标签分类表 `dev_project_tag_categories`

该表定义项目的属性维度框架，每个分类代表一个属性维度。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT (PK) | 自增主键 |
| `projectId` | INT | 所属项目ID |
| `userId` | INT | 创建者 |
| `categoryKey` | VARCHAR(50) | 分类键名（如 basic, material, function, parameter, installation, certification, special） |
| `categoryName` | VARCHAR(100) | 用户可编辑的分类显示名称 |
| `description` | TEXT | 分类说明 |
| `sortOrder` | INT | 排序序号 |
| `confirmed` | INT | 0=未确认，1=已确认 |
| `confirmedAt` | TIMESTAMP | 确认时间 |

### 1.3 项目级标签项表 `dev_project_tag_items`

该表存储每个分类下的具体标签条目，是"标签管理"功能的核心数据。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT (PK) | 自增主键 |
| `categoryId` | INT | 关联 `dev_project_tag_categories.id` |
| `projectId` | INT | 所属项目ID |
| `tagName` | VARCHAR(255) | 标签名称（中文，如"防水"、"不锈钢"） |
| `tagValue` | TEXT | 标签值/描述（可选，用于参数属性等需要值的场景） |
| `source` | ENUM('ai','manual') | 来源：AI生成或手动添加 |
| `sourceEvidence` | TEXT | 原文依据：标签来源的原文片段，用于验证标签真实性 |
| `sortOrder` | INT | 排序序号 |

## 二、业务流程全链路

整个属性标注和标签管理的业务流程分为以下阶段，形成一条完整的数据流水线：

```
数据上传 → 标签管理（定义维度框架） → 属性标注（AI打标） → 全景分析表（编辑确认） → 下游分析
```

### 2.1 阶段一：数据上传

用户在项目详情页的"数据管理"tab中上传三类Excel文件：

| 文件类型 | 关键列 | 解析函数 | 存储目标 |
|----------|--------|----------|----------|
| 搜索结果/销量数据 | ASIN、标题、品牌、价格、月销量、BSR、评论数、评分等 | `parseSalesData()` | `dev_products` 表 |
| 标题五点数据 | ASIN、标题、产品卖点、详细参数 | `parseBulletPointsData()` | `dev_products` 表（bulletPoints/specifications字段） |
| 历史月销量 | ASIN + 各月份列 | `parseHistorySalesData()` | `dev_products` 表（monthlySalesHistory JSON字段） |

三类文件的数据以ASIN为主键通过 `upsertDevProducts()` 函数合并到同一张 `dev_products` 表中。该函数采用**非破坏性更新策略**：只更新非空字段，避免不同文件上传时互相覆盖已有数据。

### 2.2 阶段二：标签管理（定义维度框架）

**入口：** 项目详情页 → "标签管理" tab → `ProjectTagManager` 组件

**初始化流程：**

1. 用户点击"初始化标签分类"按钮，调用 `devProjectTags.initCategories` 接口
2. 系统创建7个默认分类：基础分类属性、材质属性、功能属性、参数属性、安装方式、认证、特殊
3. 每个分类初始为空，等待AI生成或手动添加标签

**AI智能生成标签：**

系统提供两种AI生成方式：

| 方式 | 接口 | 说明 |
|------|------|------|
| 全量生成 | `aiGenerateTags` | 一次性为所有分类生成标签，读取项目内最多50个产品的标题和五点描述 |
| 单分类生成 | `aiGenerateCategoryTags` | 为指定分类单独生成标签，可在已有基础上补充 |

**AI生成的核心约束（反幻觉机制）：**

AI提示词包含三大硬约束，确保标签准确性：

> **绝对禁止编造** — 每个标签必须有原文依据，不在原文中出现的属性绝对不能生成。宁缺毋滥，不确定的属性宁可不输出。

AI输出要求：
- `tagName` 和 `tagValue` 必须为**中文**
- 含义相同的英文词合并为一个中文标签（如 Waterproof/Water Resistant → 防水）
- 每个标签必须附带 `evidence` 字段，标注原文依据（如"产品#3 五点: Made of premium stainless steel"）

**分层提取逻辑：**

AI分两层提取标签：
1. **第一层（通用标签）** — 多个产品共有的属性，如品类词、主流材质、通用功能
2. **第二层（差异化标签）** — 逐个产品检查独特功能、专利技术、特殊设计，确保不遗漏

**手动管理操作：**

| 操作 | 接口 | 说明 |
|------|------|------|
| 添加分类 | `addCategory` | 新增自定义属性维度 |
| 编辑分类名称 | `updateCategoryName` | 修改分类显示名称 |
| 删除分类 | `deleteCategory` | 删除整个分类及其下所有标签 |
| 添加标签 | `addTagItem` | 在指定分类下手动添加标签 |
| 编辑标签 | `updateTagItem` | 修改标签名称和值 |
| 删除标签 | `deleteTagItem` | 删除单个标签 |
| CSV批量导入 | `batchImport` | 通过CSV文件批量导入标签 |
| CSV导出 | `exportTagsCsv` | 导出所有标签为CSV（含原文依据列） |

**确认锁定机制：**

| 操作 | 接口 | 说明 |
|------|------|------|
| 确认单个分类 | `confirmCategory` | 锁定指定分类，不可编辑 |
| 解锁单个分类 | `unconfirmCategory` | 解锁指定分类，允许重新编辑 |
| 全部确认 | `confirmAll` | 一键确认所有分类 |

确认状态通过 `getTagStatus` 接口查询，返回总分类数和已确认分类数。

### 2.3 阶段三：属性标注（AI为每个ASIN打标）

**入口：** 项目详情页 → "属性标注" 阶段（在分析工作台中）

**触发条件（前置门控）：** 销量数据已确认

**执行流程：**

1. 调用 `devTagging.startTagging` 接口
2. 系统读取项目内所有产品数据（标题、五点描述、价格、品牌等）
3. 按每批5个产品分批调用LLM
4. AI为每个产品提取14个维度的标签值：产品分类、款式、主要材质、次要材质、尺寸、重量、功率、关键参数、核心功能、颜色、设计风格、安装方式、认证标准、特殊属性
5. 结果写入 `dev_product_tags` 表（产品级标签）
6. 同时更新 `dev_products.tags` JSON字段和 `tagStatus` 为 "tagged"

### 2.4 阶段四：竞品全景分析表（编辑确认）

**入口：** 项目详情页 → "全景分析表" tab → `PanoramaTable` 组件

**数据合并逻辑：**

全景分析表通过 `devPanorama.getData` 接口获取数据，合并以下来源：

| 数据源 | 来源表 | 合并方式 |
|--------|--------|----------|
| 产品基础信息 | `dev_products` | 所有字段直接展示 |
| 历史月销量 | `dev_products.monthlySalesHistory` | JSON解析为动态列 |
| 属性标签 | `dev_product_tags` | 按ASIN聚合为 `tagMap`，动态追加为标签列 |
| 标签定义 | `dev_project_tag_items` | 提供标签值下拉选项 |

**标签列编辑：**

全景表中的标签列支持手动编辑，通过 `devPanorama.updateProductTag` 接口保存。编辑方式为下拉选择（已有标签值）+ 自由输入组合。锁定后标签列不可编辑。

**确认/解锁机制：**

全景表有独立的确认状态（`dev_panorama_status` 表），确认后数据锁定不可编辑，解锁后可重新修改。

### 2.5 阶段五：下游分析（前置门控）

以下分析模块在执行前会检查全景分析表和属性标注的确认状态：

| 分析模块 | 前置条件 |
|----------|----------|
| 市场大盘分析 | 属性标注已确认 + 全景分析表已确认 |
| 属性交叉分析 | 属性标注已确认 + 全景分析表已确认 |
| 价格段分析 | 市场大盘已确认 + 全景分析表已确认 |
| 品牌竞争分析 | 市场大盘已确认 + 全景分析表已确认 |

## 三、两套标签体系的关系

当前系统中两套标签体系的关系可以概括为：

**项目级标签（标签管理）** 定义"有哪些属性维度和可选值"，相当于**字典/模板**。**产品级标签（属性标注）** 记录"每个ASIN在每个维度上的具体值"，相当于**实例数据**。

两者通过 `dimensionName` 字段关联：项目级标签的 `categoryName` 对应产品级标签的 `dimensionName`。

在全景分析表中，两者汇合展示：
- 产品级标签（`dev_product_tags`）决定每个ASIN的标签列显示值
- 项目级标签（`dev_project_tag_items`）提供下拉选择的候选值列表

在标签筛选器中，两者共同提供筛选选项，并统计每个标签值对应的产品数量。

## 六、已知问题与优化方向

### 6.1 两套标签体系的冗余

当前 `devTagging.startTagging`（属性标注）和 `devProjectTags.aiGenerateTags`（标签管理AI生成）是两个独立的AI调用，使用不同的提示词和输出格式。属性标注的结果写入 `dev_product_tags`，标签管理的结果写入 `dev_project_tag_items`。两者之间缺乏自动同步机制。

**潜在优化：** 可考虑将属性标注的AI打标逻辑与标签管理的维度框架打通，即先由标签管理定义维度，再由属性标注按照这些维度为每个ASIN打标，确保维度名称一致。

### 6.2 DevTagSettings 页面未联动

`DevTagSettings` 页面目前是静态展示，与项目级标签管理（`ProjectTagManager`）没有数据联动。可考虑将其作为全局标签模板管理，新项目可从全局模板导入维度框架。

### 6.3 标签确认与全景表确认的关系

当前标签管理的确认（`confirmCategory`）和全景表的确认（`panorama.confirm`）是独立的。下游分析的门控检查了属性标注阶段的确认状态和全景表的确认状态，但没有直接检查标签管理的分类确认状态。这意味着用户可以跳过标签管理的确认直接进入分析阶段（只要属性标注和全景表已确认）。
